# Инструкция по сдаче Security Vault

## 1. Что это за сервис

Это HTTP API сервис для хранения секретов в зашифрованном виде.

Основная идея:

- приложение не хранит секреты в коде;
- секреты записываются в SQLite;
- перед записью они шифруются;
- доступ к API ограничен ролями;
- мастер-ключ не лежит на диске в открытом виде;
- мастер-ключ восстанавливается из нескольких `child keys`;
- есть механизм `wrap/unwrap` для временной выдачи секрета по токену.

Сервис реализован на `FastAPI`.

Основные файлы:

- `main.py` - HTTP API, RBAC, маршруты;
- `secret_store.py` - логика хранилища, работа с SQLite, `seal/unseal`, `wrap/unwrap`;
- `secret_crypto.py` - криптографические примитивы проекта;
- `.env` - конфигурация;
- `docker-compose.yml` - запуск в контейнере.

## 2. Общая архитектура

Архитектура разделена на 3 слоя.

### 2.1 API слой

Файл: `main.py`

Этот слой:

- поднимает HTTP сервер;
- принимает запросы;
- валидирует входные данные через `Pydantic`;
- проверяет роль пользователя;
- вызывает методы `SecretStore`;
- переводит Python-ошибки в HTTP-ответы.

### 2.2 Storage слой

Файл: `secret_store.py`

Этот слой:

- создаёт SQLite-базу;
- хранит метаданные vault;
- хранит зашифрованные секреты;
- отвечает за `init`, `unseal`, `seal`;
- шифрует и расшифровывает секреты;
- создаёт `wrap`-токены и проверяет их.

### 2.3 Crypto слой

Файл: `secret_crypto.py`

Этот слой:

- восстанавливает мастер-ключ из `child keys`;
- создаёт verifier мастер-ключа;
- шифрует секреты;
- проверяет целостность секретов;
- подписывает `wrap`-токены;
- проверяет подпись `wrap`-токенов.

## 3. Как хранится состояние на диске

Используется SQLite с двумя таблицами.

### 3.1 Таблица `meta`

В ней лежат служебные данные:

- `status` - `sealed` или `unsealed`;
- `created_at` - время создания vault;
- `initialized_at` - время первичной инициализации;
- `last_unsealed_at` - время последнего `unseal`;
- `algorithm` - строковое описание схемы шифрования;
- `kdf` - строковое описание схемы получения мастер-ключа;
- `master_key_verifier` - verifier мастер-ключа.

Важно:

- сам `master_key` в базу не пишется;
- сами `child keys` в базу не пишутся;
- в базе лежит только verifier, по которому можно проверить, что при `unseal` восстановлен правильный ключ.

### 3.2 Таблица `secrets`

Для каждого секрета хранятся:

- `name` - имя секрета;
- `nonce` - случайное значение для шифрования;
- `ciphertext` - зашифрованный секрет;
- `tag` - MAC для проверки целостности;
- `updated_at` - время последнего обновления.

Важно:

- plaintext секрета на диск не пишется;
- даже если прочитать SQLite-файл, там будет только зашифрованное содержимое.

## 4. Как работает жизненный цикл vault

### 4.1 Состояние после старта

После запуска сервиса:

- vault находится в состоянии `sealed`;
- `master_key` в памяти отсутствует;
- читать и писать секреты нельзя;
- если приложение завершается штатно, в `lifespan` вызывается `seal()`.

### 4.2 `init`

Первичная инициализация нужна ровно один раз.

Маршрут:

- `POST /v1/init`

Что передаём:

```json
{
  "child_keys": ["key-1", "key-2", "key-3"]
}
```

Что происходит внутри:

1. Проверяется, что vault ещё не был инициализирован.
2. Из `child_keys` вычисляется `master_key`.
3. Из `master_key` вычисляется `master_key_verifier`.
4. Verifier сохраняется в `meta`.
5. Сам `master_key` остаётся в памяти процесса.
6. Vault переводится в состояние `unsealed`.

Смысл:

- сервис один раз фиксирует, какой именно набор `child keys` соответствует этому vault;
- потом этот же набор должен использоваться для повторного открытия vault.

### 4.3 `seal`

Маршрут:

- `POST /v1/seal`

Что происходит:

1. `self.master_key = None`
2. В `meta.status` записывается `sealed`.

Смысл:

- мастер-ключ удаляется из оперативной памяти;
- после этого читать и писать секреты нельзя;
- база на диске остаётся зашифрованной.

### 4.4 `unseal`

Маршрут:

- `POST /v1/unseal`

Что происходит:

1. Проверяется, что vault уже был инициализирован.
2. Из переданных `child_keys` снова вычисляется `candidate_key`.
3. Из `candidate_key` считается verifier.
4. Новый verifier сравнивается с сохранённым `master_key_verifier`.
5. Если verifier совпадает, `candidate_key` принимается как текущий `master_key`.
6. Vault переводится в `unsealed`.

Если verifier не совпал:

- возвращается ошибка `invalid child keys`;
- vault не открывается.

Смысл:

- открыть vault можно не любым набором строк, а только тем набором, который восстанавливает исходный мастер-ключ.

### 4.5 Почему это лучше, чем было раньше

Неправильный вариант:

- принять любые `child_keys`;
- детерминированно получить из них какой-то ключ;
- считать vault открытым.

Проблема:

- такой `unseal` формально проходит даже на неверных ключах;
- ошибка всплывает только позже на чтении секрета.

Текущий вариант лучше:

- неправильные `child_keys` отбрасываются сразу на этапе `unseal`;
- сервис не переходит в `unsealed`;
- поведение корректно и прозрачно для защиты.

## 5. Откуда берутся `child keys`

Это важный вопрос, который преподаватель может задать отдельно.

Ответ:

- `child keys` не генерируются автоматически сервисом;
- они не должны храниться в базе;
- они являются внешними секретами, которыми владеют операторы или администраторы.

Практически это означает:

- администратор один раз подготавливает несколько независимых секретных строк;
- при `init` они передаются сервису;
- затем эти же строки нужны для `unseal`.

Почему так:

- если сервис сам умеет полностью восстановить `child keys`, то разделение ключа теряет смысл;
- идея в том, что знание `child keys` находится вне самого vault.

## 6. Как получается мастер-ключ

Файл: `secret_crypto.py`

Функция:

- `derive_master_key(child_keys)`

Алгоритм:

1. Проверяется, что передано минимум 2 ключа.
2. Каждый ключ очищается через `strip()` и переводится в `bytes`.
3. Проверяется, что пустых ключей нет.
4. Берётся начальное значение:

```python
sha256(b"vault-master-v1")
```

5. Дальше для каждого `child_key` выполняется:

```python
digest = HMAC(digest, child_key, SHA-256)
```

6. Итоговый `digest` и есть `master_key`.

Смысл:

- итоговый ключ зависит от всех дочерних ключей;
- изменение хотя бы одного `child_key` меняет весь `master_key`.

Важно проговорить на защите:

- это учебная KDF-схема;
- для production лучше использовать более формальную схему управления ключами или threshold secret sharing;
- но для лабораторной работа выполняет требование: мастер-ключ получается из нескольких дочерних ключей.

## 7. Как работает verifier мастер-ключа

Функция:

- `derive_master_key_verifier(master_key)`

Что происходит:

1. От `master_key` считается:

```python
HMAC(master_key, b"vault-unseal-verifier-v1", SHA-256)
```

2. Результат кодируется в `base64`.
3. Это значение сохраняется в `meta.master_key_verifier`.

Зачем нужен verifier:

- он позволяет проверить правильность `unseal`, не сохраняя сам мастер-ключ;
- по verifier нельзя просто взять и использовать его как `master_key`;
- это контрольное значение для сверки.

## 8. Как работает шифрование секретов

Файл: `secret_crypto.py`

Функции:

- `encrypt(master_key, plaintext, aad)`
- `decrypt(master_key, nonce, ciphertext, tag, aad)`

### 8.1 Что такое `aad`

`aad` здесь - это additional authenticated data.

В проекте в качестве `aad` используется имя секрета:

```python
aad = record_name.encode("utf-8")
```

Смысл:

- имя секрета участвует в проверке целостности;
- нельзя незаметно подменить шифротекст одного секрета под именем другого.

### 8.2 Как происходит `encrypt`

1. Генерируется случайный `nonce` длиной 16 байт:

```python
secrets.token_bytes(16)
```

2. На основе `master_key`, `nonce` и счётчика строится псевдослучайный поток байт.
3. `plaintext` XOR-ится с этим потоком.
4. Получается `ciphertext`.
5. Для `aad + nonce + ciphertext` считается:

```python
HMAC(master_key, aad + nonce + ciphertext, SHA-256)
```

6. Результат сохраняется как `tag`.

На диск пишутся:

- `nonce`
- `ciphertext`
- `tag`

### 8.3 Как происходит `decrypt`

1. Сначала заново считается ожидаемый `tag`.
2. Он сравнивается с тем `tag`, который лежит в базе.
3. Если `tag` не совпал, выбрасывается ошибка:

- `secret integrity check failed`

4. Только если `tag` совпал, строится тот же keystream.
5. `ciphertext` XOR-ится обратно и получается исходный plaintext.

Смысл:

- если данные в базе подменили;
- если неверный `master_key`;
- если подменили `name`, `nonce` или `ciphertext`;

то целостность не сойдётся, и расшифрование не пройдёт.

### 8.4 Что важно честно сказать преподавателю

Схема учебная.

Мы используем:

- XOR-поток на базе `SHA-256`;
- отдельный `HMAC-SHA256` для целостности.

Для production правильнее было бы использовать готовую AEAD-схему:

- `AES-GCM`
- или `ChaCha20-Poly1305`

Но для лабораторной:

- секреты реально шифруются;
- целостность реально проверяется;
- данные на диске хранятся не в открытом виде.

## 9. Как работает `wrap`

Маршрут:

- `POST /v1/secrets/{name}/wrap`

Кто может вызывать:

- `admin`
- `writer`

Что передаётся:

```json
{
  "ttl_seconds": 60
}
```

Что делает `wrap_secret()`:

1. Проверяет, что vault открыт.
2. Проверяет, что `ttl_seconds > 0`.
3. Проверяет, что секрет с таким именем существует.
4. Формирует payload:

```json
{
  "secret_name": "db_password",
  "issued_at": 1710000000,
  "exp": 1710000060
}
```

5. Payload сериализуется в JSON с детерминированным порядком полей.
6. От payload считается подпись через:

```python
HMAC(master_key, payload, SHA-256)
```

7. Формируется объект:

```json
{
  "payload": {...},
  "signature": "..."
}
```

8. Весь объект кодируется в `base64` и возвращается клиенту как токен.

Идея:

- токен привязан к конкретному секрету;
- токен имеет TTL;
- токен нельзя подделать без знания текущего `master_key`.

## 10. Как работает `unwrap`

Маршрут:

- `POST /v1/unwrap`

Кто может вызывать:

- любой клиент;
- API-ключ не нужен.

Почему это безопасно:

- доступ даёт не роль, а владение валидным `wrap`-токеном;
- сам токен уже ограничен по времени и по имени секрета;
- без валидной подписи `unwrap` не сработает.

Что делает `unwrap_secret()`:

1. Проверяет, что vault открыт.
2. Декодирует base64-токен.
3. Разбирает JSON.
4. Извлекает `payload`.
5. Повторно вычисляет подпись по payload.
6. Сравнивает подпись с присланной.
7. Проверяет `exp`.
8. Если подпись и TTL корректны, получает `secret_name`.
9. Возвращает:

```json
{
  "secret_name": "...",
  "value": "...",
  "expires_at": 1710000060
}
```

Если токен просрочен:

- `token expired`

Если подпись невалидна:

- `invalid token signature`

Смысл `wrap/unwrap` в текущей архитектуре:

- `writer` или `admin` может выдать временный токен;
- другой клиент может получить секрет без собственного API-ключа;
- тем самым реализуется временная делегация доступа.

## 11. Как работает RBAC

RBAC реализован в `main.py`.

Есть 3 роли:

- `admin`
- `writer`
- `reader`

Права:

- `admin`: `status`, `init`, `unseal`, `seal`, чтение, запись, `wrap`
- `writer`: чтение, запись, `wrap`
- `reader`: чтение, список секретов, `whoami`
- `unwrap`: доступен отдельно, без API-ключа

### 11.1 Почему API-ключи в `.env` хранятся не в открытом виде

Сейчас `SECRET_API_KEYS` хранит не сами токены, а их `sha256`-хэши.

Пример:

```env
SECRET_API_KEYS='{"sha256:...":"admin"}'
```

Смысл:

- если кто-то просто прочитает `.env`, он не увидит готовый токен для использования;
- сервис при запросе берёт `X-API-Key`;
- вычисляет `sha256` от присланного токена;
- сравнивает получившийся хэш со значениями из конфига.

Это не абсолютная защита, но это лучше, чем хранить `admin-token` в открытом виде.

### 11.2 Как происходит проверка роли

Функция:

- `require_role(api_keys, allowed_roles)`

Логика:

1. Из заголовка читается `X-API-Key`.
2. От него считается `sha256`.
3. По этому хэшу ищется роль в конфиге.
4. Если хэш не найден:

- HTTP `401 invalid api key`

5. Если роль есть, но не подходит для маршрута:

- HTTP `403 role '...' cannot access this endpoint`

## 13. Демонстрационный сценарий на сдаче

Ниже лучший порядок показа.

### Шаг 1. Показать, что сервис жив

```bash
curl http://localhost:8000/health
```

Ожидание:

- `{"status":"ok"}`

### Шаг 2. Показать, что vault до инициализации закрыт

```bash
curl -X POST http://localhost:8000/v1/unseal \
  -H "Content-Type: application/json" \
  -H "X-API-Key: admin-token" \
  -d '{"child_keys":["share-1","share-2","share-3"]}'
```

Ожидание:

- ошибка `vault is not initialized`

### Шаг 3. Инициализировать vault

```bash
curl -X POST http://localhost:8000/v1/init \
  -H "Content-Type: application/json" \
  -H "X-API-Key: admin-token" \
  -d '{"child_keys":["share-1","share-2","share-3"]}'
```

Ожидание:

- `{"status":"initialized"}`

Что проговорить:

- сейчас из `child keys` вычислен `master_key`;
- в базе сохранился только verifier;
- сам `master_key` в памяти, vault открыт.

### Шаг 4. Записать секрет

```bash
curl -X PUT http://localhost:8000/v1/secrets/db_password \
  -H "Content-Type: application/json" \
  -H "X-API-Key: writer-token" \
  -d '{"value":"super-secret-password"}'
```

Что проговорить:

- секрет не кладётся на диск в открытом виде;
- в таблицу уходят `nonce`, `ciphertext`, `tag`.

### Шаг 5. Прочитать секрет

```bash
curl http://localhost:8000/v1/secrets/db_password \
  -H "X-API-Key: reader-token"
```

Что проговорить:

- чтение возможно только пока vault `unsealed`;
- при чтении сначала проверяется целостность через `tag`, потом идёт расшифрование.

### Шаг 6. Показать `seal`

```bash
curl -X POST http://localhost:8000/v1/seal \
  -H "X-API-Key: admin-token"
```

Потом:

```bash
curl http://localhost:8000/v1/secrets/db_password \
  -H "X-API-Key: reader-token"
```

Ожидание:

- ошибка `store is sealed`

Что проговорить:

- мастер-ключ удалён из памяти;
- база осталась, но расшифровать её нельзя.

### Шаг 7. Показать корректный `unseal`

```bash
curl -X POST http://localhost:8000/v1/unseal \
  -H "Content-Type: application/json" \
  -H "X-API-Key: admin-token" \
  -d '{"child_keys":["share-1","share-2","share-3"]}'
```

Ожидание:

- `{"status":"unsealed"}`

### Шаг 8. Показать некорректный `unseal`

Сначала снова сделать `seal`, потом:

```bash
curl -X POST http://localhost:8000/v1/unseal \
  -H "Content-Type: application/json" \
  -H "X-API-Key: admin-token" \
  -d '{"child_keys":["wrong-1","wrong-2","wrong-3"]}'
```

Ожидание:

- `invalid child keys`

Что проговорить:

- сервис не принимает произвольные строки;
- проверка идёт по verifier исходного мастер-ключа.

### Шаг 9. Показать `wrap/unwrap`

Сначала открыть vault правильными ключами, затем:

```bash
curl -X POST http://localhost:8000/v1/secrets/db_password/wrap \
  -H "Content-Type: application/json" \
  -H "X-API-Key: writer-token" \
  -d '{"ttl_seconds":60}'
```

Из ответа взять токен и выполнить:

```bash
curl -X POST http://localhost:8000/v1/unwrap \
  -H "Content-Type: application/json" \
  -d '{"token":"PASTE_TOKEN_HERE"}'
```

Что проговорить:

- `unwrap` не требует API-ключа;
- доступ даёт только валидный временный токен;
- токен подписан мастер-ключом и ограничен по TTL.

### Шаг 10. Показать RBAC

Например, что `reader` не может писать:

```bash
curl -X PUT http://localhost:8000/v1/secrets/forbidden \
  -H "Content-Type: application/json" \
  -H "X-API-Key: reader-token" \
  -d '{"value":"123"}'
```

Ожидание:

- `403`

И что `writer` не может смотреть статус:

```bash
curl http://localhost:8000/v1/status \
  -H "X-API-Key: writer-token"
```

Ожидание:

- `403`

## 14. Какие сильные стороны решения можно подчеркнуть

- секреты не лежат в коде;
- секреты на диске хранятся зашифрованно;
- мастер-ключ не хранится на диске;
- `seal/unseal` реализованы через несколько `child keys`;
- неверные `child keys` отбрасываются до доступа к секретам;
- есть RBAC;
- есть временная делегация доступа через `wrap/unwrap`;
- API-ключи в конфиге хранятся в виде хэшей, а не в открытом виде.

## 15. Какие ограничения надо честно озвучить

- схема шифрования учебная, а не production-grade AEAD;
- `child keys` сейчас работают по схеме "нужен весь набор", а не threshold `2-of-3`;
- SQLite подходит для лабораторной, но не для крупной распределённой системы;
- если атакующий уже выполняет код внутри процесса, одного хэширования API-ключей недостаточно.

Правильная формулировка:

"Для лабораторной цели сервис реализует безопасное хранение секретов, шифрование, контроль целостности, seal/unseal, RBAC и wrap/unwrap. Для production я бы заменил учебную криптосхему на AEAD и расширил key management."

## 16. Короткие ответы на вопросы преподавателя

### Почему `master_key` не хранится в базе?

Потому что тогда `seal/unseal` теряли бы смысл. На диске хранится только verifier, а сам ключ восстанавливается из `child keys`.

### Почему нельзя открыть vault любыми `child keys`?

Потому что при `unseal` считается verifier и сравнивается с тем, который был зафиксирован при `init`.

### Почему `unwrap` без API-ключа?

Это осознанная модель делегирования доступа: `admin` или `writer` создаёт короткоживущий токен, а другой клиент может использовать только его, без полной роли в системе.

### Почему API-ключи хранятся в `.env` как хэши?

Чтобы простое чтение `.env` не давало готовый `admin-token`. Сервис сравнивает не plaintext токены, а их `sha256`.

### Почему шифрование не AES-GCM?

Потому что это учебная лабораторная. Но данные реально шифруются, и целостность реально проверяется через `HMAC`.

## 17. Как лучше объяснить сервис одной фразой

"Это HTTP vault-сервис, который хранит секреты в SQLite в зашифрованном виде, держит мастер-ключ только в памяти, восстанавливает его из нескольких child keys, ограничивает доступ ролями и умеет выдавать временные wrap-токены для точечного доступа к секрету."
